# Validate Next.js Design Skill

이 스킬은 프로젝트 내 Next.js App Router 기반의 코드가 렌더링 최적화, 컴포넌트 분리 원칙, 그리고 데이터 보안 및 환경 변수 취급 등에서 안전하고 효율적으로 설계되었는지 검증할 때 사용합니다.

## 1. 컴포넌트 렌더링 전략 검증 (RSC vs RCC)
- **클라이언트 컴포넌트의 격리 (Isolate RCC)**: `'use client'` 지시어가 페이지 전체나 최상위 레이아웃에 남용되지 않았는지 점검합니다. 상호작용(onClick, useState)이 필요한 최소한의 말단(Leaf Node) 컴포넌트만 클라이언트 컴포넌트로 분리되었는지 확인합니다.
- **서버 컴포넌트 우선 (Prefer RSC)**: 가능하면 모든 컴포넌트가 서버 컴포넌트로 작동하고 있는지 점검하며, 클라이언트에서 불필요한 번들 사이즈를 늘리는 외부 라이브러리 사용을 지양하고 있는지 확인합니다.

## 2. 보안 취약점 및 민감 데이터 점검 (Security Checks)
- **환경 변수 노출 (Environment Variables)**: 클라이언트 컴포넌트에서 `NEXT_PUBLIC_` 접두사가 없는 중요 환경 변수(DB 비밀번호, 시크릿 키, 서비스 롤 키 등)를 참조하거나 렌더링하고 있지 않은지 엄격히 점검합니다.
- **Server Actions 접근 제어**: `src/app/actions` 등에서 정의된 Server Action들이 클라이언트에서 누구나 호출할 수 있다는 점을 인지하고 설계되었는지 확인합니다. Action 내부에서 세션 검증(Authentication) 및 권한 검사(Authorization)가 제대로 수행되고 있는지 점검합니다.
- **Supabase 인증 및 RLS 우회 여부**: 클라이언트 사이드 코드(`supabase-js`)에서 직접 중요 데이터를 조회/수정할 때 발생할 수 있는 보안 홀이 없는지, DB의 Row Level Security(RLS) 컨텍스트 하에 올바른 권한으로 작동하는지 점검합니다.

## 3. 데이터 패칭 및 아키텍처 규칙 검증
- **서버 사이드 패칭**: 클라이언트 컴포넌트 내부에서 불필요하게 `useEffect`와 `fetch`를 남용하여 데이터를 가져오는 대신, 상위 서버 컴포넌트에서 데이터를 패칭한 후 props로 내려주고 있는지 검증합니다.
- **단방향 의존성 유지**: UI 컴포넌트가 인프라스트럭처(예: DB 접근 코드)를 직접 참조하지 않고, 애플리케이션 유스케이스 또는 Server Actions를 통해서만 데이터를 주고받는지 점검합니다.

## 4. 검증 결과 리포팅 가이드라인
- 위 사항들에 대해 프로젝트 코드를 정밀하게 탐색한 뒤, **발견된 문제점(취약점)**과 **수정 제안 코드(Best Practice)**를 구분하여 명확히 설명합니다.
- 보안 관점에서 위험도가 높은(Critical) 사항은 🚨 아이콘 등을 활용하여 강조하고 최우선 수정을 권고합니다.
